# GitFlow для TaskFlow

В проекте используется стратегия ветвления, основанная на GitFlow с упрощениями для небольшой команды разработки. Она позволяет поддерживать стабильность кода, удобно разрабатывать новые функции и быстро исправлять ошибки.

## Основные ветки

- main — стабильная версия приложения. Содержит production-код, готовый к деплою. Все изменения попадают в эту ветку только через Pull Request.
- develop — основная ветка разработки. Содержит последние изменения, которые прошли ревью и готовы к тестированию.

## Вспомогательные ветки

- feature/\* — ветки для разработки новой функциональности. Создаются от develop и после завершения вливаются обратно через Pull Request.
  Примеры:
  - feature/task-board
  - feature/realtime-updates
  - feature/user-auth

- bugfix/\* — ветки для исправления ошибок, найденных во время разработки.
  Примеры:
  - bugfix/task-creation-error
  - bugfix/graphql-validation

- hotfix/\* — ветки для срочного исправления ошибок в production. Создаются от main и после исправления вливаются как в main, так и в develop.

## Правила именования веток

- Используется нижний регистр
- Слова разделяются дефисами
- Название кратко описывает задачу

Примеры:

- feature/task-filter
- feature/board-sharing
- bugfix/websocket-reconnect

## Процесс разработки

1. Создать задачу в системе управления задачами (например GitHub Issues).
2. От ветки develop создать новую ветку feature/название-функции.
3. Реализовать функциональность и сделать коммиты.
4. Запушить ветку в репозиторий.
5. Открыть Pull Request в ветку develop.
6. После прохождения код-ревью выполнить merge.
7. После накопления изменений создаётся Pull Request из develop в main для релиза.

## Коммиты

Используется стиль **Conventional Commits**.

Основные префиксы:

- feat: новая функциональность
- fix: исправление ошибки
- docs: изменение документации
- refactor: рефакторинг кода
- test: добавление тестов
- chore: служебные изменения

Примеры:

feat: добавить создание задач
fix: исправить ошибку обновления задачи
refactor: оптимизировать GraphQL резолверы

## Версионирование

Для управления версиями используется **Semantic Versioning**.

Формат версии:

vMAJOR.MINOR.PATCH

Примеры:

- v1.0.0 — первый стабильный релиз
- v1.1.0 — добавлена новая функциональность
- v1.1.1 — исправление ошибки

Каждый релиз сопровождается созданием тега в ветке main.
