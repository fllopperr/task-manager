# Техническое задание

## 1. Введение

### 1.1 Назначение документа

Данный документ представляет собой Техническое задание (Technical Specification, SRS) на разработку веб-приложения **TaskFlow** — системы управления личными и командными задачами. Он описывает **цели, требования, функциональность и ограничения проекта**, а также критерии оценки и приёмки результата.

Цель документа — зафиксировать требования, функциональные и нефункциональные характеристики, интерфейсы и ожидания от продукта так, чтобы обеспечить однозначное понимание задач разработки всеми участниками процесса (команда, заказчик, стейкхолдеры).

---

## 2. Общее описание

### 2.1 Область применения

TaskFlow предназначен для управления личными и командными задачами через веб-интерфейс. Основные сценарии включают создание, изменение, сортировку, фильтрацию задач и визуализацию статуса выполнения.

TaskFlow работает как **SPA (single-page application)**, обеспечивает быстрый отклик на действия пользователя, а также поддерживает базовую и расширенную модель (freemium) использования.

### 2.2 Характеристика пользователей

Пользовательские группы:

- Студенты
- Junior / Middle разработчики
- Малые команды
- Люди, ведущие личные задачи

Эти категории пользователей не требуют сложной корпоративной функциональности, однако ожидают **интуитивный UX**, лёгкую навигацию и стабильную работу интерфейса — особенно при частых изменениях данных.

### 2.3 Ограничения

TaskFlow должен:

- Работать в современных браузерах;
- Быть адаптивным (mobile-first);
- Обеспечивать устойчивость интерфейса под высокой нагрузкой.

## 3. Функциональные требования

### 3.1 Управление пользователями

- Регистрация через email/пароль;
- Авторизация с сохранением сессии;
- Восстановление пароля;
- Возможность переключения между аккаунтами.

---

### 3.2 Управление задачами

- Создание, редактирование, удаление задач;
- Перемещение задач (Drag & Drop);
- Задание сроков, приоритетов, меток;
- Фильтрация и поиск по задачам.

---

### 3.3 Управление досками

- Создание и удаление досок;
- Группировка задач по колонкам;
- Экспорт/импорт данных (опционально для Pro).

---

## 4. Нефункциональные требования

### 4.1 Требования к производительности

- Среднее время отклика UI не более 200 мс;
- Обработка действия Drag & Drop без задержек на клиенте;
- Поддержка одновременной работы нескольких тысяч пользователей.

---

### 4.2 Требования к UX/UI

- Простой, интуитивный интерфейс;
- Минимум экранов для выполнения ключевых задач;
- Лёгкий старт без обучения.

---

### 4.3 Требования к безопасности

- HTTPS для всех соединений;
- Защита от XSS/CSRF;
- Безопасное хранение паролей (hash).

---

## 5. Интерфейсы

### 5.1 Внешние интерфейсы

#### Интерфейсы пользователя
- Панель навигации;
- Список досок;
- Канбан-доска;
- Панель фильтрации и поиска.

#### Интерфейсы API
- REST/GraphQL API для взаимодействия с backend;
- Обработчик изменений состояния задач (Realtime, если предусмотрено).

### 5.2 Архитектура приложения
TaskFlow реализуется как клиент-серверное SPA с выделением трех основных слоёв:

- **Frontend** — SPA, отвечает за интерфейс, клиентскую бизнес-логику и управление состоянием;
- **Backend** — REST/GraphQL API, управление задачами, пользователями и досками;
- **Слой кэширования и realtime-коммуникации** — обеспечивает производительность и мгновенную синхронизацию.

---

### 5.3 Кэширование и Realtime

#### Redis
Используется для:

- кэширования часто запрашиваемых данных (доски, задачи);
- хранения сессий пользователей;
- временного хранения промежуточных состояний;
- Pub/Sub для синхронизации между WebSocket-соединениями.

#### WebSocket
Используется для:

- синхронизации изменений задач между клиентами;
- мгновенного обновления состояния досок;
- отображения изменений без перезагрузки страницы.

**Примечание:** Redis усиливает WebSocket, но для MVP можно начать только с WebSocket, а Redis подключить при росте нагрузки.

## 6. Критерии приёмки

Продукт будет считаться выполненным, если:
- Все функциональные сценарии реализованы и протестированы;
- Проект соответствует описанным нефункциональным требованиям;
- UI/UX соответствует утверждённым макетам;
- Производительность соответствует KPI.

---

## 7. Тестирование

Тестирование включает:
- Unit-тесты для функций;
- Интеграционные тесты для UI и API;
- E2E-тесты основных сценариев;
- Производительное тестирование под нагрузкой.
